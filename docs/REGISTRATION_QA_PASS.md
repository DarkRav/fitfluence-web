# QA pass: регистрация и onboarding UX

Дата проверки: 2026-02-28

## Технические проверки

1. `npm run typecheck` — успешно.
2. `npm run build` — успешно.
3. `npm run lint` — успешно, без ошибок.

Остаточные предупреждения lint:

- `@next/next/no-img-element` в media-компонентах (предсуществующие, не относятся к флоу регистрации/onboarding).

## Проверка ключевых сценариев

### A) Новый пользователь

Ожидаемое поведение:

1. Регистрация в Keycloak.
2. Возврат в web-admin.
3. `/v1/me` определяет отсутствие профиля.
4. Переход в `/onboarding` (или сразу на целевую форму при одном required-флаге).
5. Создание профиля.
6. Повторный `/v1/me`.
7. Редирект в корректный home.

Статус: реализовано в коде через единый `resolvePostLoginPath`.

### B) Существующий influencer

Ожидаемое поведение:

- После login/callback -> `/v1/me` -> `/influencer/programs`.

Статус: реализовано.

### C) Существующий admin

Ожидаемое поведение:

- После login/callback -> `/v1/me` -> `/admin/programs`.

Статус: реализовано.

### D) Athlete при отсутствии athlete web UI

Ожидаемое поведение:

- После создания athlete profile -> `/athlete/created`.
- Страница: сообщение о мобильном приложении + действия «Выйти» и опционально «Создать профиль инфлюэнсера».

Статус: реализовано.

## Дополнительные проверки UX

1. Экран логина:

- Есть чёткое разделение «Войти» и «Создать аккаунт».
- Есть объяснение, что регистрация происходит в Keycloak.

2. `/onboarding`:

- Автопереход на нужную форму, если required только один профиль.
- Экран выбора остаётся доступным для ручного входа.

3. `returnTo`:

- Поддержан безопасный возврат после login/onboarding для внутренних маршрутов.
- Невалидные пути игнорируются с fallback в role-based home.

## Итог

Целевой UX-флоу регистрации и onboarding приведён к детерминированной модели:

- один источник решения (`/v1/me`),
- единая стратегия редиректов,
- отсутствие технических промежуточных экранов в пользовательском сценарии,
- понятные русские тексты и действия на каждом шаге.
