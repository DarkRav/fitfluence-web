# UX-аудит регистрации и онбординга

## Контекст

Цель: убрать трение в первых шагах пользователя после логина через Keycloak и сделать переход к рабочему разделу предсказуемым.

Оценивались сценарии:

1. Новый пользователь после регистрации в Keycloak.
2. Пользователь без доменного профиля.
3. Пользователь с одним профилем.
4. Пользователь с двумя профилями.

## Основные UX-проблемы

### 1) Непредсказуемые редиректы после логина

- Проблема: в нескольких местах была дублирующая логика переходов (`/login`, `/auth/callback`, `/welcome`, `route-guard`, формы onboarding).
- Симптом: пользователь мог попасть на `/me` или другую промежуточную страницу без понимания «что делать дальше».
- Влияние: рост когнитивной нагрузки и ощущение «сломанных» переходов.

### 2) Дебажный экран `/me` использовался как продуктовый landing

- Проблема: после входа пользователь видел техническое JSON-представление профиля.
- Влияние: интерфейс выглядел административно-техническим, а не пользовательским.

### 3) Смешение входа и регистрации на login-экране

- Проблема: кнопка «Создать аккаунт» не объясняла, что регистрация происходит во внешнем Keycloak с возвратом в приложение.
- Влияние: ощущение потери контекста и «непонятно куда меня уводит».

### 4) Entry onboarding не учитывал серверные флаги

- Проблема: пользователь всегда видел выбор роли, даже когда `/v1/me` уже определил нужный профиль.
- Влияние: лишний шаг и дублирование решения.

### 5) Недостаточно контекста на выборе роли

- Проблема: карточки роли были короткими и не объясняли результат выбора.
- Влияние: повышалась вероятность ошибочного выбора.

### 6) Формы onboarding были функциональными, но не «быстрыми»

- Проблема: слабое разделение обязательного/необязательного, мало guidance по следующему шагу, неявный сценарий возврата.
- Влияние: замедление прохождения onboarding.

### 7) Ошибки и тупиковые состояния

- Проблема: часть ошибок была сформулирована технически; не везде были явные действия восстановления.
- Влияние: фрустрация, повторные попытки без понимания причины.

### 8) Нет единой политики `returnTo`

- Проблема: после логина/онбординга пользователь не всегда возвращался к исходной задаче.
- Влияние: терялся контекст действия, особенно при deep-link переходах.

## Решения (продуктовые)

1. Единый bootstrap после логина/callback:

- Всегда запрашивать `/v1/me`.
- Если onboarding required -> `/onboarding`.
- Иначе -> role-based home (`/influencer/programs`, `/admin/programs`, `/athlete/created`).

2. Убрать `/me` из пользовательского маршрута:

- Оставить `/me` только как внутренний/сервисный экран, не как post-login landing.

3. Пересобрать login UX:

- Чёткие CTA: «Войти» и «Создать аккаунт».
- Короткий microcopy: регистрация происходит в защищённом окне Keycloak, затем возврат в Fitfluence.

4. Оптимизировать `/onboarding`:

- Если требуется только athlete или только influencer профиль — автоматический переход на соответствующую форму.
- Экран выбора показывать только когда выбор действительно нужен.

5. Усилить карточки выбора роли:

- Добавить понятное «для кого» и «что получите после создания профиля».

6. Формы onboarding:

- Единая структура: заголовок, краткое объяснение, компактные поля, inline-ошибки, один главный CTA.
- Явно отметить необязательные поля.

7. Стандартизировать error/empty/loading состояния:

- Понятные русские тексты.
- Понятное действие восстановления (повторить, вернуться к выбору, выйти).

8. Добавить безопасный `returnTo`:

- Возврат только на разрешённые внутренние пути.
- При невалидном `returnTo` использовать role-based home.
- Если onboarding ещё нужен — сначала onboarding, потом возврат.

## Ожидаемый эффект

- Меньше лишних шагов на пути «логин -> первый полезный экран».
- Предсказуемое поведение редиректов.
- Понятный выбор роли и более быстрый onboarding.
- Снижение числа тупиковых состояний после регистрации.
